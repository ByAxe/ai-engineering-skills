# Quarkus Project Model and Build-Time Semantics

Use this as the entry point for every Quarkus task. Quarkus is not just a Jakarta runtime with fast startup: augmentation, indexing, generated bytecode, configuration phases, extension metadata, and native compilation make repository context part of program semantics.

## 1. Profile the project before changing code

Record evidence for:

- Quarkus platform/BOM and plugin version
- Maven or Gradle wrapper version and module boundaries
- configured Java release/toolchain and CI/runtime JDK
- installed extensions and whether the REST/persistence stack is blocking or reactive
- JVM-only, container, native, or mixed deployment target
- dev/test/prod profiles and config sources
- generated sources, OpenAPI/client generation, annotation processors, and migrations
- deployment modules or custom extensions

Useful commands, adapted to the repository:

```bash
./mvnw -q help:effective-pom
./mvnw -q dependency:tree
./mvnw quarkus:info
./gradlew dependencies
./gradlew quarkusInfo
python3 scripts/profile_java_project.py . --format markdown
```

Do not assume every command exists. Prefer the checked-in wrapper and inspect plugin tasks/goals first.

## 2. Platform alignment

Quarkus extensions are normally version-aligned through the platform/BOM. Before adding or upgrading an extension:

1. inspect the imported platform/BOM
2. search the effective dependency graph for an existing capability
3. confirm the extension artifact for that Quarkus generation
4. avoid pinning an independent extension version unless project policy requires it
5. compile immediately after the smallest build-file edit

A plausible Maven coordinate generated from memory is not evidence. Do not add a dependency merely to satisfy an invented import.

## 3. Augmentation changes the mental model

Quarkus performs substantial work during the build:

- indexes application/dependency types
- discovers beans and endpoints
- validates injection and configuration
- transforms bytecode and generates classes
- records initialization logic
- removes eligible unused beans
- prepares metadata for JVM and native execution

Consequences for agents:

- code that looks unused may be referenced through annotations, generated metadata, observers, service loading, serialization, or extension build steps
- a configuration or bean choice may be fixed into the artifact at build time
- local dev mode can regenerate and hot-reload things that a packaged artifact cannot
- changing annotations may alter generated code and startup behavior even when source-level call sites do not change
- a classpath-only mental compilation is insufficient; run a Quarkus-aware build gate

## 4. Build-time versus runtime configuration

Classify a relevant property before promising it can be changed after deployment:

- **build-time fixed:** read during augmentation and encoded into the artifact
- **build-and-run-time fixed:** participates in both phases under extension rules
- **runtime:** can vary when the built application starts

The exact classification is extension- and version-specific. Check the project’s Quarkus config reference or generated config documentation rather than inferring from the property name.

### Failure pattern

An agent adds a runtime environment variable for a build-time property, tests only in dev mode, and claims production will switch behavior without rebuilding.

### Required check

Build the artifact once, run it with the intended runtime override, and prove the observable behavior. For native output, test the native executable separately.

## 5. Profiles are not interchangeable

Distinguish:

- build profile used during augmentation
- runtime profile used when the packaged application starts
- test profile selected by the test harness
- Maven/Gradle profiles, which are a separate build-tool concept

Build-profile bean conditions such as `@IfBuildProfile` are resolved while building; changing only the runtime profile does not necessarily change bean resolution. Do not use profile conditions as a general-purpose runtime feature-flag system.

## 6. Indexing and discovery

Quarkus ArC uses simplified bean discovery and an index. Dependencies may need:

- a bean archive descriptor
- a Jandex index
- explicit `quarkus.index-dependency` configuration
- extension-provided discovery rules

Before adding a scope annotation to random third-party types, understand why the type is not discovered. Prefer indexing or an explicit producer/adapter when that matches the boundary.

Do not delete an index configuration entry as “obsolete” until injection, reflection, serialization, and custom extension behavior are checked.

## 7. Generated sources and source of truth

Common generated surfaces include:

- REST/OpenAPI clients and DTOs
- protobuf, Avro, GraphQL, or schema-generated types
- Hibernate metamodels
- MapStruct and annotation-processor output
- Quarkus-generated bytecode/classes
- test resources and Dev Services configuration

Never edit `target/`, `build/`, or generated source output directly. Find the schema, template, plugin configuration, source annotation, or extension code that owns it. Regenerate and review the resulting diff.

## 8. Dev mode is not packaged mode

Dev mode may provide:

- continuous testing
- hot reload
- Dev Services
- Dev UI
- relaxed development defaults
- local service discovery or ephemeral credentials

A feature that works in `quarkus:dev` can still fail in:

- packaged JVM mode
- container runtime
- a minimal production image
- native mode
- a production profile without Dev Services

Use the narrowest packaged-mode check relevant to the change. Never encode a Dev Services hostname, port, credential, or lifecycle assumption into production logic.

## 9. Extension code versus application code

Custom Quarkus extensions normally separate deployment-time and runtime concerns. Do not move ordinary application code into build steps merely for performance. When touching extension code:

- preserve deployment/runtime module boundaries
- avoid loading application classes incorrectly during augmentation
- understand build items, recorders, synthetic beans, and class initialization
- run extension tests across augmentation and runtime behavior
- verify native implications

This skill focuses primarily on application engineering. For extension internals, consult the exact Quarkus extension-authoring guide for the project version.

## 10. Native is a distinct compatibility target

Native compilation uses a closed-world analysis. Risks include:

- reflection or dynamic proxies not visible to analysis
- resources not included in the image
- classpath scanning at runtime
- dynamic class loading
- serialization metadata
- JNI and platform libraries
- static initialization that captures time, randomness, environment, sockets, or secrets at build time
- charset, locale, security provider, and URL protocol assumptions

Do not respond by registering whole packages reflexively. Identify the exact access path, use extension/framework registration where possible, and validate the produced executable.

## 11. Quarkus change protocol

For each Quarkus change:

1. **Profile:** version, extension, execution model, config phase, target mode.
2. **Trace:** entry point through CDI, transaction, persistence/client call, and response/event.
3. **Classify:** source-time, augmentation-time, startup-time, request-time, or shutdown-time behavior.
4. **Patch narrowly:** avoid concurrent framework or dependency upgrades.
5. **Verify progressively:** unit/component test, `@QuarkusTest`, packaged integration, native only where required.
6. **Report:** name the tested mode. “Tests pass” is incomplete when only dev-mode behavior was exercised.

## High-signal project-model smells

| Signal | Likely risk | Check before fixing |
|---|---|---|
| extension versions individually pinned | platform drift | effective dependency management and compatibility policy |
| `javax.*` imports in modern Quarkus code | generation mismatch | project Quarkus/Jakarta generation and migration state |
| runtime override of a build-time property | ineffective deployment change | property phase and packaged-mode behavior |
| source edits under `target/generated-sources` | overwritten patch | generator/source schema |
| application relies on Dev Services address | production failure | production config and service lifecycle |
| classpath scanning/reflection added casually | native failure/startup cost | exact registration and target mode |
| feature works only in dev mode | packaging/config gap | packaged JVM and relevant profile |
| unindexed dependency bean assumed discoverable | unsatisfied injection | index/discovery rules |
| broad `@RegisterForReflection` | image bloat and hidden design issue | precise reflective consumer and extension support |

## Exit checklist

- [ ] Quarkus/BOM/plugin versions came from the repository.
- [ ] Added APIs and properties exist for that version.
- [ ] Build-time/runtime behavior was classified.
- [ ] Generated and indexed surfaces were respected.
- [ ] Dev Services assumptions did not leak into production.
- [ ] The tested deployment mode is named.
- [ ] Native behavior was tested when it is a production contract.
